                  6 Decompile the Kernel with Ghidra

                  Can we actually see the RISC-V Code inside the Linux Kernel?

                  Yep! Let‚Äôs decompile the Armbian Kernel with Ghidra, the popular tool for Reverse Engineering‚Ä¶

                  1. In Ghidra, create a New Project

                  2. Click File > Import File

                  3. Select boot/vmlinuz-5.15.0-starfive2 and enter these Import Options‚Ä¶

                    Format: Raw Binary

                    Language: RISCV > RV64GC (RISCV:LE:64:RV64GC:gcc)

                    (StarFive JH7110 has 4 √ó RV64GC U74 Application Cores)

                    (RV64GC is short for RV64IMAFDCZicsr_Zifencei)

                    Options > Base Address: 0x40200000

                    (Based on the U-Boot Configuration from above)

                    (Ghidra thinks it‚Äôs PE Format because of ‚ÄúMZ‚Äù‚Ä¶ But it‚Äôs not!)

                    (TODO: Pic should show 0x4020 0000 instead)

                  4. In the Ghidra Project, double-click vmlinuz-5.15.0-starfive2

                    Analyse the file with the Default Options.

                  Wait a while and we‚Äôll see the Decompiled Linux Kernel in Ghidra‚Ä¶

                  At Address 0x4020 0002 we see a Jump to FUN_402010c8.

                  Double-click FUN_402010c8 to see the Linux Boot Code‚Ä¶

                  The CSR Instructions look interesting, but we‚Äôll skip them today.

                  When we match the RISC-V Instructions, the Linux Kernel Source File is probably this‚Ä¶

                  • linux/arch/riscv/kernel/head.S

                  The first RISC-V Instruction looks kinda sus‚Ä¶

                  // Load -13 into Register S4
li  s4,-0xd

// Jump to Actual Boot Code
j   FUN_402010c8

                  It‚Äôs highly sus because the First Instruction doesn‚Äôt do anything meaningful!

                  Remember the ‚ÄúMZ‚Äù at the top of our Kernel Image?

                  For Legacy Reasons, the Linux Kernel embeds ‚ÄúMZ‚Äù to signify that it‚Äôs a PE / COFF File, to look like a UEFI Application.

                  The RISC-V Instruction li assembles into Machine Code as ‚ÄúMZ‚Äù. That‚Äôs why it‚Äôs the first instruction in the Linux Kernel!

                  We‚Äôll recreate ‚ÄúMZ‚Äù in our NuttX Kernel.

                  (‚ÄúMZ‚Äù refers to Mark Zbikowski)

                  (Linux Kernel pretends to be a DOS File‚Ä¶ NuttX Kernel pretends to be Linux. Hilarious!)

                  8 Appendix: Decode the RISC-V Linux Header

                  What‚Äôs inside the RISC-V Linux Header?

                  Earlier we downloaded the Armbian Kernel Image‚Ä¶

                  • ‚ÄúArmbian Image for Star64‚Äù

                  Let‚Äôs decode the RISC-V Linux Header at /boot/Image. (Pic above)

                  (Which is sym-linked to /boot/vmlinuz-5.15.0-starfive2)

                  We dump the bytes in the file‚Ä¶

                  hexdump vmlinuz-5.15.0-starfive2

                  Which will begin with this RISC-V Linux Image Header‚Ä¶

                  • ‚ÄúBoot Image Header in RISC-V Linux‚Äù

                  Here are the decoded bytes‚Ä¶

                  1. code0: Executable code

                    (4 bytes, offset 0x00)

                    4D  5A  6F  10

                  2. code1: Executable code

                    (4 bytes, offset 0x04)

                    60  0C  01  00

                  3. text_offset: Image load offset, little endian

                    (8 bytes, offset 0x08)

                    00  00  20  00  00  00  00  00

                  4. image_size: Effective Image size, little endian

                    (8 bytes, offset 0x10)

                    00  C0  56  01  00  00  00  00

                  5. flags: Kernel flags, little endian

                    (8 bytes, offset 0x18)

                    00  00  00  00  00  00  00  00

                  6. version: Version of this header (MinL MinM . MajL MajM)

                    (4 bytes, offset 0x20)

                    02  00  00  00

                  7. res1: Reserved

                    (4 bytes, offset 0x24)

                    00  00  00  00

                  8. res2: Reserved

                    (8 bytes, offset 0x28)

                    00  00  00  00  00  00  00  00

                  9. magic: Magic number, little endian, ‚ÄúRISCV\x00\x00\x00‚Äù

                    (8 bytes, offset 0x30)

                    52  49  53  43  56  00  00  00

                  10. magic2: Magic number 2, little endian, ‚ÄúRSC\x05‚Äù

                    (4 bytes, offset 0x38)

                    52  53  43  05

                  11. res3: Reserved for PE COFF offset

                    (4 bytes, offset 0x3C)

                    40  00  00  00

                  Our NuttX Kernel shall recreate this RISC-V Linux Image Header. (Total 0x40 bytes)

                  (Or U-Boot Bootloader might refuse to boot NuttX)

                  Why is the Image Load Offset set to 0x20 0000?

                  Image Load Offset (from Start of RAM) is set to 0x20 0000 because the Linux Kernel boots at 0x4020 0000.

                  The Image Load Offset is hardcoded in the Linux Kernel Boot Code for 64-bit RISC-V‚Ä¶

                  #ifdef CONFIG_RISCV_M_MODE
  /* If running at Machine Privilege Level... */
  /* Image Load Offset is 0 MB from Start of RAM */
  .dword 0

#else
  #if __riscv_xlen == 64
    /* If running at 64-bit Supervisor Privilege Level... */
    /* Image Load Offset is 2 MB from Start of RAM */
    .dword 0x200000

  #else
    /* If running at 32-bit Supervisor Privilege Level... */
    /* Image Load Offset is 4 MB from Start of RAM */
    .dword 0x400000
  #endif
#endif

                  (Source)
