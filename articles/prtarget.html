                  Pros and Cons of the new implementation?

                  (1) New Implementation is Safer: We don‚Äôt use pull_request_target, and we don‚Äôt checkout the Entire NuttX Repo based on the PR. So we‚Äôll never execute any Malicious Code submitted in the PR. (2) New Implementation is Quicker: It‚Äôs faster since we don‚Äôt checkout the entire repo. Also pr-size-labeler actually runs in a Docker Container, we don‚Äôt need that any more. (3) But it might be quirky under Heavy Load. Remember that workflow_run trigger will write the PR Labels as a Second Job? When we run out of GitHub Runners, the PR Labels might never be applied. The Build Logic in arch.yml will execute a Complete NuttX Build if it can‚Äôt find the PR Labels. (4) Will the Build Workflow be triggered too early, before the workflow_run trigger? Hopefully not. The Build Workflow begins in the Fetch-Source stage, checking out the Entire Repo and uploading everything in 1.5 minutes, followed by the Select-Builds stage (arch.yml) reading the PR Labels. Before 1.5 minutes, rightfully our workflow_run trigger would have written the PR Labels to the PR.

                  Are we reimplementing EVERYTHING from the Official GitHub Labeler actions/labeler?

                  We won‚Äôt implement the Entire GitHub Labeler actions/labeler, just the bare minimum needed for NuttX. We‚Äôre emulating the Labeler Config .github/labeler.yml as is, because someday GitHub might invent a Secure Way to Label PRs inside pull_request_trigger. (Then we‚Äôll switch back to the Official GitHub Labeler actions/labeler) There‚Äôs something really jinxed about the way GitHub designed PR Labeling in pull_request_trigger, it‚Äôs a terrible security hack. I‚Äôll write an article about this someday :-)
